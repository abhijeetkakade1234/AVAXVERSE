## 2025-03-11 - DoS via Push Payments in Escrow Split Resolution
**Vulnerability:** The `resolveDisputeSplit` function in `Escrow.sol` used direct `.call{value: ...}("")` with `require(ok)` for multi-party fund distribution. This created a severe Denial of Service (DoS) vulnerability: if one party (e.g., the client or freelancer) was a smart contract that reverted on receiving ETH, the entire transaction would revert, permanently locking the funds for all parties and breaking the dispute resolution process.
**Learning:** Even if one party is malicious or misconfigured, it should not break the functionality for others. The push payment pattern (sending funds directly within the transaction and reverting on failure) is fundamentally unsafe for multi-party distributions, particularly when dealing with untrusted actors.
**Prevention:** Always use a pull-over-push pattern for multi-party fund distributions. In Solidity, this means transferring funds using a safe wrapper (like `_safeTransfer`) that catches reverts and credits a `pendingWithdrawals` mapping, allowing users to withdraw their funds independently in separate transactions.
## 2026-03-08 - Added Security Headers
**Vulnerability:** The application was missing standard HTTP security headers, leaving it potentially vulnerable to clickjacking and MIME-type sniffing.
**Learning:** Added security headers in `next.config.ts` to enforce Strict-Transport-Security, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, and X-DNS-Prefetch-Control.
**Prevention:** Always include a robust set of security headers in the Next.js configuration to mitigate common web vulnerabilities.
## 2025-03-14 - Gas Griefing DoS in _safeTransfer
**Vulnerability:** In both `Escrow.sol` and `EscrowFactory.sol`, the internal `_safeTransfer` function used `.call{value: amount}('')` to transfer ETH to users as a part of a pull-over-push pattern implementation. Although it safely handled reverts by crediting `pendingWithdrawals`, it did not restrict the gas forwarded. An attacker (or a misconfigured contract) could implement a fallback/receive function containing an infinite loop or high gas consuming operations. This would consume all gas allocated to the transaction (gas griefing), causing the entire transaction to revert out-of-gas, preventing the `pendingWithdrawals` fallback from ever executing and locking other functionalities (like job cancellation or dispute resolution).
**Learning:** Even when a `.call` gracefully catches failure booleans, forwarding all available gas (`gas()` default behavior) allows the callee to grief the transaction by hitting the block gas limit or out-of-gas errors.
**Prevention:** Always restrict the amount of gas forwarded in external calls when transferring ETH to potentially untrusted addresses (e.g., `.call{value: amount, gas: 50000}('')`). This ensures that even if the callee consumes all forwarded gas, the parent transaction retains enough gas to catch the error, execute the fallback logic, and complete successfully.
